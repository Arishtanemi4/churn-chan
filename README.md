The SaaS Simulator: Technical Architecture Guide for Frontend Engineering
This backend is a concurrent, multi-agent simulation engine. It is designed to take hypothetical SaaS upgrade scenarios (pricing plans, user personas, and targeted marketing prompts) and run them through a local Large Language Model (LLM) to predict how real users would react.

As the frontend engineer, you don't need to worry about the internal orchestration or the LLM logic. You only need to understand the Inputs (what you send us) and the Outputs (what we send back).

1. The Core Interaction Model: JSON In, JSON Out
Our entire system communicates exclusively via JSON. The API is entirely stateless—we do not store your data in a database. You send a complete configuration payload, the backend processes it in memory, and returns the calculated results.

The User Journey (from your perspective):

Configuration: The user uses your UI to define a subscription plan, select/edit user personas, and write upgrade prompts.

Execution: Your frontend packages this state into a massive JSON object (the SimRequest) and POSTs it to the /simulate endpoint.

Processing (The Black Box): The backend receives your JSON. It spawns dozens of parallel, asynchronous tasks. For every combination of Persona and Prompt, it feeds a structured scenario into the local LLM, forcing the AI to return a rigid JSON decision (upgrade, wait, or leave) and a text explanation.

Aggregation: The backend tallies the raw AI decisions, calculates percentages, computes the expected revenue uplift, and selects representative text explanations.

Rendering: The backend returns a final, aggregated JSON array (the SimResponse). Your UI maps over this array to render the Results Dashboard tables and charts.

2. The Data Contract (What You Send & What You Get)
The data structures are strictly enforced by Pydantic on the backend. Here is the blueprint for the primary endpoint: POST /simulate.

The Input: SimRequest Payload
This is the single JSON object you must construct and send when the user clicks "Run Simulation."

plan (Object): The baseline environment. Contains the name (string), the usage limit (integer), and the price (float).

personas (Array of Objects): The test subjects. Each object defines a user. If you pass an archetype string (e.g., "freelancer_designer"), our backend will automatically fetch the complex psychological traits (price sensitivity, churn risk) for that persona. If the user creates a custom persona via sliders in your UI, you must pass the exact price_sensitivity (float 0.0 - 1.0) and churn_risk (float 0.0 - 1.0) values.

prompts (Array of Objects): The messaging interventions. Each object contains an id (string), the actual marketing text (string), and a trigger condition (string, e.g., "at 100% usage limit").

runs_per_combo (Integer, Optional): Determines the statistical sample size. Important: Higher numbers increase accuracy but slow down the response time, as it multiplies the total LLM calls.

The Output: SimResponse Payload
This is the aggregated JSON array you will receive back. It is pre-calculated and ready to be mapped into your UI tables.

The payload contains a results array. Each object in this array represents the outcome of one specific combination of a Persona and a Prompt.

persona (String): The name of the persona tested (e.g., "Alex").

prompt_id (String): The ID of the prompt tested. Use this ID to map the result back to the specific prompt text in your UI state.

upgrade_pct / wait_pct / leave_pct (Floats): The statistical distribution of the AI's decisions, expressed as decimals (e.g., 0.66 means 66%). These three numbers will always sum to exactly 1.0.

expected_uplift (Float): The projected revenue generated by this specific combination. Formula: (upgrade_pct * plan.price).

sample_reasons (Array of Strings): Up to three distinct text explanations generated by the LLM detailing why it made its decision (e.g., "The plan is too expensive for my current needs."). Render these below your main charts as qualitative insights.

3. Support Endpoints
Beyond the main simulation engine, the backend provides two simple GET endpoints to support your UI setup.

GET /health: Use this endpoint for status indicators or to ensure the API is reachable before allowing the user to click "Run." It simply returns {"status": "ok"}.

GET /personas: Hit this endpoint when your application mounts. It returns the dictionary of predefined, hardcoded archetypes (e.g., the freelancer, the startup engineer, the procurement manager) along with their default traits and descriptions. Use this data to populate the preset dropdowns or pre-fill the sliders in your Persona Editor UI.
